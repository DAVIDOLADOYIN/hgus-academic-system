# Stage 6 Brief — Exports & Admin Overview

**Status:** Pre-development planning document
**Depends on:** Stages 1–5 fully deployed

---

## What Stage 6 Delivers

Three capabilities that complete the admin and output side of the system:

1. **Results Overview screen** — replaces the stub `broadsheetHome` tile on the Admin Panel with a real school-wide completion dashboard. Admins can see the status of every class group at a glance and drill into any class's broadsheet without going through the teacher-facing Broadsheet menu.

2. **Broadsheet PDF & Excel export** — adds download buttons directly to the broadsheet table screen. One button generates a formatted PDF; a second generates a proper `.xlsx` file the school can open in Excel or Google Sheets.

3. **Subject Scores PDF export** — adds a Download PDF button to the per-subject component scores screen (broadsheetSubjectDetail). Produces a printable score sheet showing every student's C/W, ASS, ATT, Test1, Test2, and Exam column for one subject.

4. **Assignment Carry-Forward** — when Admin opens Manage Assignments and no Teacher-Subject Assignments exist for the current term/session, the system detects this automatically and presents a one-tap carry-forward prompt showing exactly what would be copied from the previous term. Admin can confirm or dismiss. Without this, score entry is silently blocked at every term change.

## Feature 1 — Results Overview Screen

### What it shows

A single screen listing every class group (JSS 1, JSS 2, JSS 3, SSS 1, SSS 2, SSS 3) for the **active session and term**. Each row shows:

- Class group name (e.g. "SSS 2")
- Score completion percentage: how many of the required (students × subjects × 6 components) cells have been filled in
- A visual status badge for each of the three locks:
  - **Scores** — percentage bar + fraction (e.g. "126 / 180 entries")
  - **Broadsheet** — ✓ Ready / ✗ Locked
  - **Results** — ✓ Ready / ✗ Locked
- A "View Broadsheet" button that is active only when broadsheetUnlocked = true

Tapping "View Broadsheet" takes the Admin directly to the broadsheet table for that group (same as the teacher-facing flow but bypassing the lock menu — Admins can already view everything from the broadsheet home).

### Where it fits in the UI

The Admin Panel (`adminHome`) already has a tile:
```
{ icon: '📊', label: 'Results Overview', sub: 'Broadsheets & result slips', view: 'broadsheetHome' }
```

In Stage 6, this tile's `view` value changes from `'broadsheetHome'` to `'resultsOverview'`. The new `resultsOverview` view is a proper screen. The existing `broadsheetHome` view remains unchanged — teachers still use it to access their class groups.

### Server endpoint needed

`serverGetResultsOverview(token)` — returns completion data for all class groups for the active term/session. Calls `CompletionService.getClassGroupCompletion()` for each group and bundles the results. Returns:

```
{
  session: '2025/2026',
  term: 'First Term',
  groups: [
    {
      groupKey: 'JSS 1',
      broadsheetUnlocked: false,
      resultsUnlocked: false,
      detail: {
        activeStudentCount: 38,
        subjectCount: 18,
        scores: { required: 4104, entered: 2160, complete: false },
        psq:    { studentsRequired: 38, studentsComplete: 0, complete: false },
        remarks:{ studentsRequired: 38, studentsComplete: 0, complete: false }
      }
    },
    ...  // one entry per group
  ]
}
```

### How groups are derived

`serverGetResultsOverview` derives the list of group keys from `SheetService.getAllClasses()`, applies the same `getGroupKey_()` stripping logic used throughout (removes department suffix from SSS class names), and deduplicates. This gives exactly the 6 groups (JSS 1–3, SSS 1–3) without any hardcoding.

## Feature 2 — Broadsheet PDF & Excel Export

### Where the buttons appear

On the existing `broadsheetView` screen (the all-subjects totals table), a sticky export bar is injected below the table — the same pattern as the PDF action bar in Stage 5. It contains two buttons:

- **↓ Download PDF** — downloads a formatted A4 PDF of the broadsheet table
- **↓ Download Excel** — downloads a `.xlsx` file

Both buttons are always shown when the broadsheet is visible (broadsheetUnlocked is already confirmed to get here).

### Broadsheet PDF content

The PDF mirrors what is on screen. Content:

- **Orientation:** Landscape A4 (`@page { size: A4 landscape; }` in the inline `<style>` block)
- **Header:** School logo (top-left) + school name + "Broadsheet" title + class group name, term, session
- **Table:** Student Name | per-subject Total columns | Average | Grade | Position
- **Footer (bottom of every page):** Generated on: `dddd, dd mmmm yyyy` at `HH:mm` | Generated by: `[Staff ID]`

Styled with inline CSS (same constraint as result slip PDFs — no CSS variables).

Uses `Utilities.newBlob(html, 'text/html').getAs('application/pdf')` — no additional GAS services needed.

The `staffId` of the currently logged-in user is passed into the server function via the token (already resolved by `AuthService.validateToken(token)` at the start of every endpoint).

### Broadsheet Excel content

The `.xlsx` is generated by creating a temporary Google Spreadsheet in GAS, populating it with the broadsheet data, exporting it as XLSX via `DriveApp.getFileById(id).getAs('application/vnd.openxmlformats-officedocument.spreadsheetml.sheet')`, then immediately trashing the temporary spreadsheet.

`DriveApp` is a built-in GAS service — no Advanced Services need to be enabled.

The spreadsheet structure per class sheet tab:

- **Row 1:** School logo image (inserted via `sheet.insertImage(logoBlob, 1, 1)`) + school name in a merged header cell
- **Row 2:** "Broadsheet — [class] — [term] — [session]" in a merged subtitle cell
- **Data rows:** Student Name | Subject A | Subject B | … | Average | Grade | Position
- **Final row:** "Generated on: [dddd, dd mmmm yyyy HH:mm] by [Staff ID]" in a merged footer cell

One sheet tab per class in the group (e.g. SSS 1 has up to 3 tabs: Art, Commerce, Science).

Filename pattern: `SSS_1_Broadsheet_First_Term_2025-2026.xlsx`

### Server endpoints needed

- `serverGetBroadsheetPDF(token, classGroupKey)` — returns `{ base64Pdf, filename }`
- `serverGetBroadsheetExcel(token, classGroupKey)` — returns `{ base64Excel, filename }`

Both call `BroadsheetService.getBroadsheetForClassGroup()` (already exists) to get the data, then pass it to a new `ExportService`.

**Important:** These endpoints must enforce the broadsheet lock — return `{ success: false, code: 'LOCKED' }` if `broadsheetUnlocked` is false. This prevents someone bypassing the UI.

## Feature 3 — Subject Scores PDF Export

### Where the button appears

On the existing `broadsheetSubjectDetail` screen (the per-subject component score table), a sticky export bar is injected below the table with a single button:

- **↓ Download Score Sheet**

### Content

The PDF is a printable class score sheet for one subject and one term:

- **Header:** School logo (top-left) + school name + subject name + class group name, term, session
- **Table:** Student Name | C/W | ASS | ATT | Test1 | Test2 | Exam | Total | Grade — one row per active student
- **Signature block (bottom of table):** A clearly bordered section with two side-by-side cells:
  - Left cell: "Teacher's Signature: ________________________" + a blank line for the date
  - Right cell: "Administrator's Signature: ________________________" + a blank line for the date
- **Footer:** Generated on: `dddd, dd mmmm yyyy` at `HH:mm` | Generated by: `[Staff ID]`

Uses the same HTML → PDF approach as above (inline styles, no CSS variables). Portrait orientation (default).

### Server endpoint needed

- `serverGetScoreSheetPDF(token, classGroupKey, subjectId)` — returns `{ base64Pdf, filename }`

Reads scores via `SheetService.getScoresForClassGroup()` (already exists) and student list via `SheetService.getCachedStudents()`.

Filename pattern: `SSS_1_Mathematics_Scores_First_Term_2025-2026.pdf`

## Feature 4 — Assignment Carry-Forward

### The problem it solves

At the start of each new term, Teacher-Subject Assignments must exist for teachers to enter scores. Currently if an Admin sets a new term in Session Settings, all teachers immediately lose access to score entry — there are no assignments for the new term. The Admin must manually recreate every assignment row.

### How it works

When the Admin opens **Manage Assignments** (`assignments` view), the system checks whether any Teacher-Subject Assignments or Form Master Assignments exist for the **currently active term and session**.

- If assignments exist → show the normal assignment management screen (no change to existing behaviour).
- If no assignments exist → show a Carry-Forward Prompt screen instead of the normal list.

### The Carry-Forward Prompt screen

Displayed in place of the assignments list when no assignments exist for the active term/session. Shows:

- A banner: "No assignments found for [Term] [Session]."
- A preview of what would be carried forward: a grouped list of all Form Master and Teacher-Subject Assignments from the most recent prior term/session that has data.
- Two action buttons:
  - **Carry Forward All** — copies all assignments from the prior period into the current term/session. New rows written to both Form Master Assignments and Teacher-Subject Assignments sheets. After completion, the normal assignments screen is shown.
  - **Start Fresh** — dismisses the prompt and shows the normal (empty) assignment management screen so Admin can build assignments manually.

### What "copy" means for each assignment type

**Form Master Assignments:**
- New row with the same StaffId + ClassId + Session + GrantedFullAccess
- `IsActive = TRUE`, `ActiveFrom = today`
- Assignment ID auto-generated

**Teacher-Subject Assignments:**
- New row with the same StaffId + ClassId + SubjectId
- Term and Session updated to the current active term/session
- Assignment ID auto-generated

### Prior term detection

The system looks backwards from the current term/session to find the most recent period that has at least one Teacher-Subject Assignment row. The order to check:

```
Current: Second Term 2025/2026
Check 1: First Term 2025/2026      ← look here first
Check 2: Third Term 2024/2025      ← then previous session
Check 3: Second Term 2024/2025
... (stop after finding any data, or after 6 attempts)
```

If no prior assignments are found at all, the prompt says "No prior assignments found to carry forward" and only the "Start Fresh" button is shown.

### Server endpoints needed

- `serverGetCarryForwardPreview(token)` — checks whether carry-forward is needed and returns the prior assignments for preview. Returns `{ needed: bool, fromTerm, fromSession, formMasterAssignments: [], teacherAssignments: [] }`.
- `serverExecuteCarryForward(token)` — writes the copied assignment rows and returns the new assignments list.

## Logo Configuration

All PDF and Excel exports show the school logo in the header. The logo is stored in Google Drive and referenced by a single file ID in `Config.gs`. No base64 strings are hardcoded anywhere.

### How to set it up (one-time, before deploying Stage 6)

1. Upload the school logo image (PNG or JPG, recommended size: 200×80px or similar letterhead proportion) to any location in Google Drive.
2. Right-click the file → **Get link** → copy the file ID from the URL. The file ID is the long string between `/d/` and `/view` in the shareable link.
3. Open `Config.gs` in the GAS editor and add this constant (it will be added by Stage 6):

```javascript
const SCHOOL_LOGO_FILE_ID = 'your-google-drive-file-id-here';
```

4. The file must be accessible to the GAS script. Since the script runs under the same Google account that owns the file, this is automatic — no sharing settings needed.

### How it works at generation time

`ExportService.gs` fetches the logo like this:

```javascript
var logoBlob   = DriveApp.getFileById(SCHOOL_LOGO_FILE_ID).getBlob();
var logoBase64 = Utilities.base64Encode(logoBlob.getBytes());
var logoMime   = logoBlob.getContentType(); // e.g. 'image/png'
var logoDataUri = 'data:' + logoMime + ';base64,' + logoBase64;
```

For **PDF exports**: the data URI is embedded directly in the `<img src="...">` tag inside the generated HTML. The GAS PDF renderer fully supports base64-embedded images.

For **Excel exports**: the raw blob is passed to `sheet.insertImage(logoBlob, 1, 1)` which places the image at column 1, row 1 of the header area.

### Fallback if no logo is configured

If `SCHOOL_LOGO_FILE_ID` is empty or the Drive file cannot be fetched, the export continues without the image — the school name text header is shown instead. No error is thrown; the generation never fails over a missing logo.

## Constraints & Notes

**No new GAS services to enable:**
- Broadsheet PDF uses `Utilities.newBlob().getAs('application/pdf')` — built-in, same as Stage 5.
- Broadsheet Excel uses `DriveApp.getFileById().getAs(xlsxMimeType)` — `DriveApp` is a built-in GAS service; no Advanced Services required. A temporary spreadsheet is created, populated, exported, and immediately trashed.
- Score sheet PDF uses the same HTML → PDF method.
- Logo fetching uses `DriveApp.getFileById(SCHOOL_LOGO_FILE_ID).getBlob()` — same built-in DriveApp, no extra services.

**One new constant in Config.gs:**
- `SCHOOL_LOGO_FILE_ID` — Google Drive file ID of the school logo. Set this before deploying Stage 6. If left empty, exports render without the logo (text header only).

**CSS variables cannot be used inside PDF/Excel HTML:**
- All inline `style="..."` attributes in `ExportService.gs` must use plain color literals (e.g. `#1565C0`, `#333333`), not `var(--blue-800)`. This is the same rule applied in `PDFService.gs` (Stage 5).

**Results Overview is read-only for Admins:**
- The overview screen shows completion state but does not allow editing. Clicking "View Broadsheet" navigates to the standard `broadsheetView` screen (which already exists and works).

**Carry-Forward affects only assignments — not scores:**
- Executing carry-forward copies Form Master Assignment rows and Teacher-Subject Assignment rows. It never touches the Scores, PSQ, Remarks, or Student Term Status sheets.

**Stage inheritance rule:**
- `StyleBase.html` and `Code.gs` must be built by copying the Stage 5 versions verbatim and only appending new additions — never rewriting from scratch.
